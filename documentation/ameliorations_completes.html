Liste Complete des Ameliorations de PmaControl

  • Genere le: 2026-03-12 01:42:25
  • Objectif: backlog d’amelioration transverse, couvrant architecture, securite, tests, DX, performance et gouvernance

1. Architecture et code

  1. Decouper les gros controleurs (Cleaner, Schema, Aspirateur, Backup, Dot3) en services orientés domaine.
  2. Sortir la logique metier des actions web vers des classes applicatives testables.
  3. Introduire des DTO pour les entrees API, formulaires et jobs.
  4. Remplacer les acces directs aux superglobales par une couche request explicite.
  5. Formaliser les contrats de sortie HTML/JSON/fichier.
  6. Unifier le style de code et les conventions de nommage.
  7. Supprimer le code mort, les commentaires historiques et les chemins de debug temporaires.
  8. Centraliser la construction d’URLs et de routes.
  9. Introduire des exceptions metier dediees par domaine.
  10. Documenter explicitement les dependances entre controleurs et bibliotheques.

2. Base de donnees et acces SQL

  1. Remplacer toute interpolation SQL par requetes preparees.
  2. Introduire un repository layer pour les entites critiques.
  3. Normaliser les transactions dans les parcours d’ecriture.
  4. Ajouter des contraintes BD manquantes la ou le metier les suppose deja.
  5. Distinguer clairement referentiel, time series, caches et journaux.
  6. Rationaliser les tables ts_* et documenter leur retention.
  7. Ajouter des index verifies par usage sur les tables de topologie, backup et monitoring.
  8. Mettre en place une strategie de migrations coherentement versionnee.
  9. Mesurer et limiter la croissance des tables system-versioned.
  10. Mieux separer les ecritures techniques des donnees de gouvernance.

3. Securite

  1. Ajouter une protection CSRF globale.
  2. Renforcer la validation des entrees route, GET et POST.
  3. Centraliser les checks d’autorisation metier.
  4. Isoler les appels shell et interdire la concatenation brute.
  5. Basculer les secrets vers un vault ou un stockage externe.
  6. Reduire les privileges des comptes MySQL utilises par l’application.
  7. Segmenter les cles SSH par cas d’usage.
  8. Journaliser toutes les operations sensibles avec acteur, cible et resultat.
  9. Filtrer les logs pour eviter la fuite de secrets et chemins sensibles.
  10. Revoir l’exposition des endpoints de telechargement et d’import.

4. Testabilite et qualite

  1. Ajouter des tests unitaires sur les services extraits des gros controleurs.
  2. Ajouter des tests d’integration sur API, backup, restore et topologie.
  3. Mettre en place des fixtures SQL minimales et reutilisables.
  4. Ajouter une base de tests dediee ou un environnement ephemeral.
  5. Introduire un pipeline CI pour lint, phpunit et verification doc.
  6. Ajouter une analyse statique PHPStan/Psalm graduellement.
  7. Couvrir les parcours d’erreur et d’autorisation.
  8. Tester explicitement les workflows CLI.
  9. Ajouter des snapshots de graphes et de documents generes.
  10. Mettre en place une strategie de non-regression sur les exports.

5. Performance et exploitation

  1. Identifier les requetes les plus couteuses des ecrans lourds.
  2. Mettre en cache les vues purement analytiques.
  3. Deplacer les taches longues hors requete web.
  4. Industrialiser la gestion des fichiers pivots, locks et md5.
  5. Documenter les timeouts reseau par integration.
  6. Ajouter des garde-fous de concurrence sur workers et cleaner.
  7. Rendre observable la file de jobs et les echecs techniques.
  8. Ajouter des budgets de retention sur logs, archives et historiques.
  9. Segreguer les volumes filesystem temporaires et definitifs.
  10. Prevoir une politique de reprise sur incident pour backup et collecte.

6. UX operateur et documentation

  1. Uniformiser les messages d’erreur et de succes.
  2. Ajouter une doc fonctionnelle par domaine, pas seulement par classe.
  3. Expliquer dans l’UI les impacts des operations destructives.
  4. Ajouter un guide operateur pour backup, cleaner, schema, cluster.
  5. Generer un catalogue de routes/action par controleur.
  6. Ajouter un glossaire metier des objets mysql_*, ts_*, dot3_*, backup_*.
  7. Documenter les prerequis systeme exacts par module.
  8. Ajouter des exemples JSON/cURL pour toutes les integrations.
  9. Produire une matrice ACL lisible par role.
  10. Documenter les parcours d’onboarding et de support.

7. Modernisation technique

  1. Introduire progressivement des types retour PHP natifs.
  2. Eliminer les var et proprietes dynamiques legacy.
  3. Ajouter strict_types=1 la ou c’est viable.
  4. Remplacer les utilitaires statiques les plus couplants par des services injectes.
  5. Isoler la couche Glial legacy derriere des adapters.
  6. Reduire le nombre de points d’entree qui ecrivent en base depuis le web.
  7. Factoriser les wrappers HTTP/SSH/SQL.
  8. Ajouter une strategy de compatibilite PHP documentee.
  9. Standardiser le packaging des scripts CLI.
  10. Simplifier la relation entre docs generees et docs hand-written.

8. Ordre recommande de chantier

Lot 1

  • SQL, CSRF, secret management, shell hardening

Lot 2

  • extraction services sur Api, Server, Client, Environment, Tag, Alias, StorageArea

Lot 3

  • refonte par domaine des blocs lourds Backup, Cleaner, Aspirateur, Schema, Dot3

Lot 4

  • CI/CD, static analysis, test data management, observabilite

9. Conclusion

La liste d’ameliorations est volontairement exhaustive et structurante. Le projet n’a pas besoin d’une simple couche cosmetique; il a besoin d’une mise a niveau par domaines, en commencant par la securite et les gros noeuds de complexite, puis en consolidant la testabilite et la gouvernance documentaire.
